# ⚙️ القرارات المعمارية (ADRs)
## Architecture Decision Records

> كل قرار معماري كبير يُوثَّق هنا بصيغة: السياق → القرار → البدائل المرفوضة → النتائج المتوقعة.

---

## ADR-001: اختيار sentence-transformers بدل قواعد بيانات شعاعية سحابية

- **📋 السياق:** نريد بحثاً دلالياً (semantic) إلى جانب FTS5، مع الحفاظ على فلسفة "خفيف، PyYAML فقط، بلا API".
- **✅ القرار:** استخدام **sentence-transformers** محلياً (نموذج `all-MiniLM-L6-v2`، ~80 MB، 384-بُعد)، وتخزين الـ embeddings في SQLite (BLOB أو امتداد `sqlite-vec`).
- **📊 البدائل المرفوضة:**
  - *Milvus / Pinecone / Weaviate:* تتطلب خادماً/API/اشتراكاً → تكسر الخفّة والاستقلالية.
  - *OpenAI Embeddings API:* يكلّف مالاً ويتطلب اتصالاً دائماً + يربطنا بمزوّد (يخالف Swappable Brain).
- **🎯 النتائج المتوقعة:** بحث دلالي محلي مجاني، ملف واحد، يعمل offline. التبعية وراء **tier optional** مع lazy import (لا تُحمَّل إلا عند تفعيل Hybrid).
- **القاعدة الذهبية المرتبطة:** 3 (تبعيات مبرّرة).

## ADR-002: CodeAct كأداة إضافية لا بديلة

- **📋 السياق:** Manus يستخدم CodeAct (كود Python كفعل موحّد). عندنا 7 أدوات ثابتة تعمل جيداً.
- **✅ القرار:** إضافة `code_act_executor.py` كأداة **ثامنة إضافية**، مع إبقاء الأدوات الـ7 كما هي.
- **📊 البدائل المرفوضة:**
  - *استبدال الأدوات الـ7 بالكامل بـ CodeAct:* يكسر التوافق العكسي + يخالف القاعدة 7 (لا تحذف الأصلي).
  - *عدم إضافة CodeAct إطلاقاً:* يفوّت مرونة تركيب الأدوات في خطوة واحدة.
- **🎯 النتائج المتوقعة:** الوكيل يختار: أداة بسيطة للمهام البسيطة، أو CodeAct للمهام المركّبة. التنفيذ محصور داخل `_safe_path`/sandbox، يلتقط stdout/stderr كـ Observation.
- **القاعدة الذهبية المرتبطة:** 2 (الأمان)، 7 (لا حذف).

## ADR-003: تفعيل sort_keys=True في json.dumps

- **📋 السياق:** `agent.py` يستخدم `json.dumps(action, ensure_ascii=False)` بلا `sort_keys`. ترتيب المفاتيح غير الحتمي يكسر استقرار بادئة الـ KV-cache.
- **✅ القرار:** إضافة `sort_keys=True` لكل استدعاءات `json.dumps` المتعلقة بالسياق.
- **📊 البدائل المرفوضة:**
  - *الإبقاء على الحالي:* يهدر فرصة توفير ~10× في تكلفة الـ tokens المخزّنة.
- **🎯 النتائج المتوقعة:** بادئة JSON حتمية → نسبة إصابة cache أعلى → تكلفة أقل وسرعة أعلى. تحسين رخيص وفوري (درس Manus الأول).
- **القاعدة الذهبية المرتبطة:** 1 (يعمل مع كل الأدمغة — تحسين عام).

## ADR-004: backend الـ sandbox قابل للتبديل (local افتراضي)

- **📋 السياق:** نريد تشغيلاً سحابياً (Daytona/E2B) دون فرضه على كل المستخدمين أو كسر الخفّة.
- **✅ القرار:** `tools/cloud_sandbox.py` كـ backend اختياري؛ الإعداد `sandbox.backend` يقبل `local` (افتراضي) / `daytona` / `e2b`. `daytona-sdk` وراء tier optional.
- **📊 البدائل المرفوضة:**
  - *فرض Daytona دائماً:* يتطلب حساباً سحابياً + يكسر التشغيل المحلي البسيط.
  - *عدم دعم السحابة:* يفوّت Hibernation و Zero-Trust isolation.
- **🎯 النتائج المتوقعة:** نفس الكود يعمل محلياً وسحابياً بتغيير سطر إعداد واحد (يحاكي فلسفة Swappable Brain).
- **القاعدة الذهبية المرتبطة:** 1 (مرونة)، 2 (عزل)، 3 (تبعية مبرّرة).

## ADR-005: فهرسة الطبقة الأولى فقط للمهارات (Progressive Disclosure)

- **📋 السياق:** 717 مهارة. تحميلها كاملة في السياق مستحيل (إغراق tokens).
- **✅ القرار:** فهرسة **الطبقة 1 فقط** (name + description، ~80 token/مهارة) في جدول البحث، وتحميل الطبقة 2 (الجسم الكامل) للمهارة المختارة فقط، والطبقة 3 (scripts) عند الطلب.
- **📊 البدائل المرفوضة:**
  - *فهرسة الجسم الكامل لكل مهارة:* 717 × 2000 token = إغراق + تكلفة هائلة.
  - *تجاهل المهارات:* يفوّت 717 مهارة احترافية.
- **🎯 النتائج المتوقعة:** اكتشاف ذكي خفيف + تحميل عند الحاجة فقط. يطابق معيار Anthropic للمهارات.
- **القاعدة الذهبية المرتبطة:** 3 (خفّة).

## ADR-006: الإبقاء على urllib بدل requests في brain.py

- **📋 السياق:** `brain.py` يستخدم `urllib` (stdlib) للاتصال بالنماذج، لا `requests`.
- **✅ القرار:** الإبقاء على `urllib` — لا نضيف `requests` كتبعية.
- **📊 البدائل المرفوضة:** *إضافة requests:* تبعية غير ضرورية تخالف فلسفة "PyYAML فقط".
- **🎯 النتائج المتوقعة:** صفر تبعيات شبكية إضافية. الكود يبقى نظيفاً ومستقلاً.
- **القاعدة الذهبية المرتبطة:** 3.
